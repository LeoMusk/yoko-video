# 可行性调研待办

> 在动手写 M1 代码前，需要先把这些问题调研清楚。每一项都可能改变项目设计。建议你筛源清单的同时，我们并行推进这里的几个高优问题。

---

## P0：信息源可行性（最关键）

### Q1：RSS 源测活
- 对 `sources-draft.md` 中所有标记"官方 RSS"的源逐个测试
- 检查项：URL 是否 200、是否最近 30 天有更新、字段是否完整（title / link / content / pubDate）
- 输出：`sources_status.md`，每个源标记 ✅ 活 / ⚠️ 半活（更新慢）/ ❌ 死

### Q2：RSSHub 路由可用性
- GitHub Trending、机器之心、量子位、即刻、知乎、微信公众号等都依赖 RSSHub
- **核心风险**：RSSHub 公共实例频繁挂掉，自建需要长期维护
- 决策点：
  - (a) 用 [rsshub.app](https://rsshub.app) 官方公共实例（稳定性中等，免费）
  - (b) 用第三方付费实例（如 RSSHub Pro，月费几十元）
  - (c) 自建 RSSHub（一次性 Docker 部署 + 维护成本）
  - (d) 关键路由自己写脚本爬（更脆弱）
- 调研：跑通 (a) 看实际成功率，再决定

### Q3：Twitter / X 怎么办
- 这是 Tier 3 的核心问题（见 `sources-draft.md`）
- 调研项：
  - X API Basic 实际能拉到多少推文/天，是否够用
  - socialdata.tools / TweetScout 等第三方实际定价、合规情况
  - Nitter 现状（公开实例存活率、自建可行性）
- 决策点：先选 30 个目标账号，估算每天总推文量，对照各方案的配额和成本

### Q4：微信公众号接入
- 已知方案：WeRSS、wxnmp、自建（基于公众平台的"订阅号助手"）
- 合规：微信对自动化抓取的态度
- 决策点：选 1 个方案试 10 个公众号 1 周，看稳定性

### Q5：抖音 / 视频号 / YouTube 同行监控
- 用于看"同赛道哪些视频火了"，反向找选题
- 候选工具：飞瓜数据、新榜、TikTok Creative Center
- 这块**未必属于 M1**，可能作为独立的"竞品分析"模块或第三方工具，需评估

## P0：M2 选题评分 prompt 设计

> M2 是项目 IP 所在，prompt 设计直接决定输出质量。

### Q6：重要性评分的维度有哪些
候选维度：
- **新颖性**：相对于过去 7 天是否是新信息
- **首发性**：源头权威性（OpenAI 官博 > 转发 > 二次解读）
- **影响范围**：可能影响多少人 / 行业（一个 SOTA 模型 vs 一个小工具）
- **可视频化**：是否容易做成 60-180s 短视频（一个数字 vs 一个复杂概念）
- **赛道契合度**：是否符合用户已发视频的风格 / 关注点
- **同行覆盖度**：中文同行是否已经讲过（讲过的就不抢，没讲的是机会）

### Q7：怎么避免 LLM 评分的常见 bias
- "AI 名词堆砌偏好"（更倾向给提到 GPT/Claude 的内容打高分）
- "热度偏好"（推荐已被多源报道的内容，反而错过早期信号）
- "保守偏好"（不敢给奇怪/边缘的内容高分，但爆款常出在这里）

调研方法：用历史数据回测——找过去 30 天用户认为该做视频的事件，反推 prompt 能否选出它们。

### Q8：评分单元是单条还是聚合
- 单条评分：每个 raw item 独立打分，省 token
- 聚合评分：把跨源讲同一事件的内容合并再打分，更准但工程量大
- 决策点：M2 大概率需要"先聚类、后排序"两步

## P1：M3 脚本模板设计

### Q9：怎么把用户的"表达风格"喂给 LLM
- 用户已发视频的脚本 / 文案，是否有结构化版本可用？
- 至少需要 10-20 条样本做 few-shot
- 调研：用户能否提供这些样本（导出已发视频字幕 / 文案）

### Q10：脚本输出该结构化到什么程度
候选：
- **完全自由 markdown**：LLM 自由发挥（不便于后续自动化）
- **半结构化**：每段标注 [口播] / [B-roll 关键词] / [字幕 highlight]（适合数字人）
- **完全结构化 JSON**：每个分镜带时长、文案、视觉描述（适合自动化但 LLM 输出不稳定）

MVP 阶段建议半结构化即可。

### Q11：60-180s 短视频的脚本工程经验
- 黄金前 3 秒怎么写
- 信息密度（每秒承载的信息量）
- "钩子—爆点—展开—收口"的结构是否值得固化
- 调研建议：找 30 条用户认为"自己看了能学到东西"的同赛道爆款视频，逆向拆解结构

### Q13：PPT 信息流 / 智能成片

- 是否支持脚本驱动（输入文案自动配画面 + 字幕）
- 是否有 API（大概率没有，目前可能是手动操作）

## P2：法务与平台风险

### Q14：信息搬运 + 解读的版权边界
- 搬运论文摘要 + 自己解读：OK
- 搬运他人 Newsletter 内容：高风险
- 搬运他人视频 / 截屏：高风险
- 调研：哪些常见做法属于"二次创作 / 合理引用"，哪些属于侵权

### Q15：账号平台风险
- 抖音 / 视频号对"AI 生成内容"的标注要求
- 数字人内容的限流情况
- "搬运"内容的限流 / 封号风险

---

## P2：数据与状态管理

### Q16：每日 raw items 是否长期保存
- 不保存：M2 重跑 / 回测都不可能
- 全部保存：30 天就是几十 GB
- 折中：原始数据保留 30-90 天，过滤后的"高分选题"长期归档

### Q17：用户操作历史是否需要回流
- 用户挑了哪些选题、舍弃了哪些 → 可用于改进 M2 评分（隐式反馈）
- MVP 阶段先记录，不一定要做反馈学习

## 调研优先级建议

按"对项目设计影响大 × 可以现在就做" 排序：

1. **Q1（RSS 测活）** — 1-2 小时能搞定，结果直接决定 M1 范围
2. **Q3（Twitter 方案）** — 这个决定整个 Tier 3 是否存在
3. **Q9（用户已发视频样本）** — 这个决定 M3 能不能做出"像本人"的脚本
4. **Q2（RSSHub）** — 影响接入复杂度
5. **Q12（视频 SaaS 选型）** — Week 2 才需要，但提前调研不亏
6. 其他 — 看 MVP 跑通后再细化
